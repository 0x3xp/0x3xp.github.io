Abstract

This post documents the reconnaissance and static analysis phase of an attack surface investigation into the Windows DirectComposition (DComp) subsystem. The work was performed against a live Windows 10 x64 host (Build 19045.6466) as preparation for kernel-mode fuzzing of win32kbase.sys.

The goal was clear: extract System Service Numbers (SSNs) for the DComp channel syscalls, trace the full user-mode commit pipeline from application to kernel boundary, and understand how the kernel deserializes and dispatches command batches on the other side of that boundary. No exploitable vulnerability was found during this engagement — but the structural knowledge gained is a direct prerequisite for any meaningful fuzzing campaign against this subsystem.

// RESEARCH STATUS
Analysis is complete. No CVE was identified in this phase. The next steps section outlines the fuzzing harness work required to proceed toward vulnerability discovery. All research conducted on an isolated local machine with no network access.

Three primary activities are covered:

  • Enumeration of NtDComposition* symbols from win32u.dll using WinDbg and extraction of exact SSN values from disassembled syscall stubs
  • Static reverse-engineering of dcomp.dll via Ghidra tracing the full user-mode commit pipeline through three distinct phases
  • Kernel-side analysis of NtDCompositionCommitChannel in win32kbase.sys — pointer probing, handle resolution, and batch dispatch logic

Methodology

The investigation was conducted in two parallel tracks. The first was dynamic analysis — attaching WinDbg to a live user-mode process and querying the win32u.dll export table. Since win32u.dll is the thin user-mode wrapper for all Win32k syscalls, its stubs contain the literal SSN values loaded into EAX immediately before the syscall instruction fires. Disassembling those stubs gives us the exact numbers needed to invoke the kernel directly.

The second track was static analysis via Ghidra. dcomp.dll was loaded with its PDB and the decompiler was used to trace the commit pipeline from the high-level CDevice::Commit entry point — the function called by every DirectComposition application to flush pending visual changes — all the way down to the NtDCompositionCommitChannel syscall stub. The kernel binary win32kbase.sys was then loaded separately to reverse the Ring 0 handler.

3.3 — SSN Summary & Technical Observations

FunctionSSN (Hex)Subsystem
NtDCompositionCommitChannel0x111FWin32k
NtDCompositionCreateChannel0x1124Win32k

mov r10, rcx — Mandatory x64 Syscall Convention. This is an architectural requirement, not a DComp quirk. The syscall instruction overwrites RCX with the return address (for sysret), so the first argument — which the calling convention places in RCX — must be saved to R10 first. Every Win32k syscall stub begins this way.

Win32k SSN Range. Standard NT Executive syscalls occupy the 0x0000–0x01FF range. These IDs are in the 0x11xx block, confirming they route through the Win32k graphical subsystem dispatcher rather than the core NT executive. This is a different dispatch table, different filtering mechanisms, and a historically more vulnerable attack surface.

Practical note: The 0x11xx Win32k range is significant for EDR bypass research. Many kernel callback filters are registered for NT syscalls but have weaker coverage of Win32k-specific syscalls. Direct invocation at these SSNs can bypass some user-mode hook chains entirely.

Phase 1 — Entry Point: CDevice::Commit

CDevice::Commit is the transactional flush entry point. Before any kernel work happens, it performs several housekeeping operations:

  • Synchronization — Acquires either a CRITICAL_SECTION via EnterCriticalSection or a vtable-dispatched lock at this+0x08, depending on whether a custom lock object exists at this+0x48.
  • Dirty-object rebuild — Iterates a singly-linked list at this+0x1c8, calling CRebuildableObject::Rebuild on each pending visual update before committing.
  • Virtual surface optimization — If g_VSAllowOptimizer is set, invokes CVirtualSurfaceOptimizer::OptimizeVirtualSurface to prune surfaces idle longer than 999ms and reclaim memory.
  • Back-pressure handling — If the kernel batch queue is saturated, spins on CommitToKernel + WaitForCommitCompletionInternal until the queue drains.
  • Telemetry — Under an SRW lock, invokes CTelemetryHelper::LogTelemetryLocked if 600 seconds have elapsed since the last telemetry flush.

Phase 2 — Batch Preparation: CDevice::CommitToKernel

CommitToKernel validates the batch before it crosses the privilege boundary. The most interesting part for fuzzing purposes is the integrity check — if the committed batch size doesn't match the expected counter, the process calls FailFastForKernelBatchFailure rather than propagating the mismatch. This is a deliberate hardening decision to prevent corrupted batches from reaching the kernel.

The batch size integrity check at this+0xa0 is the gate. A fuzzing harness that bypasses dcomp.dll entirely via direct syscall avoids this check. The kernel's own validation at the Ring 0 boundary becomes the target. That is the objective of the next phase of this research.

Kernel-Side Analysis — NtDCompositionCommitChannel

The kernel handler for this syscall is located at offset 0x5eb90 in win32kbase.sys, export index 1367. This is where Ring 0 takes control.

5.1 — User-Mode Pointer Validation

The first thing the kernel does is probe every user-supplied pointer against MmUserProbeAddress. This is mandatory defensive coding for any kernel function that accepts user-mode memory references — attempting to dereference a pointer above MmUserProbeAddress without probing first would be a kernel vulnerability itself.

The probe pattern works by substituting the pointer with MmUserProbeAddress if it falls outside user space, then performing a read — the structured exception handler fires on an invalid user address, cleanly returning STATUS_ACCESS_VIOLATION rather than crashing the kernel.

5.2 — Process and Channel Object Lookup

After pointer validation, the kernel resolves the calling process's Win32k state via PsGetCurrentProcessWin32Process(), then looks up the channel object in a generic table keyed by the handle passed from user space.

This lookup is lock-protected: KeEnterCriticalRegion disables normal APC delivery, and ExAcquireResourceExclusiveLite takes exclusive ownership of the channel table lock. A missing handle returns STATUS_ACCESS_DENIED (0xC0000022) rather than a NULL dereference.

5.3 — Batch Deserialization Loop

The core of the handler. Once the channel object is resolved and ref-counted, the kernel enters a loop that dequeues and dispatches command batches until the channel's ring buffer is empty. Each iteration calls FUN_1c005ee20 (the batch dispatcher, documented in 5.4) and passes the dequeued batch to FUN_1c005f594 for compositor dispatch.

5.4 — Batch Command Dispatcher: FUN_1c005ee20

This is the most complex function in the kernel-side analysis. It contains the fast path / full path branch logic, the command serializer chain, and ETW telemetry hooks.

Fast Path vs Full Path. The fast path is taken when all dirty-state flags at offsets 0x30, 0xf0, and the linked lists at 0x1a8–0x250 are clear. If clean and param_4 == 0, the function simply wakes the compositor event at this+0x168 and returns — no batch record is constructed.

Command Serializer Chain. The full path chains through a sequence of specialized serializers, each responsible for a specific resource type:

SerializerResource Type
FUN_1c00600c4Visual transforms
FUN_1c00657fc / 1c0065744 / 1c0065684 / 1c00655f4Surface bindings & hit-test regions
FUN_1c005f34cComposition targets
FUN_1c005f2acCross-adapter resources
FUN_1c0060000 / 1c005ff8c / 1c005fef0Synchronization objects

Special Command Records. Two fixed-format records are emitted at the end of the serializer chain under specific conditions:

  • TYPE_0x34 — Control block record, emitted if this+0x2e8 is non-null. Serializes 9 DWORD fields from offsets 0x2f0–0x310.
  • TYPE_0x0C — Visibility change record, emitted if this+0x250 is 1 or 2. Signals window show/hide state to the compositor.
// FUZZING SURFACE — HIGH INTEREST
The command type IDs (0x34, 0x0C) and the serializer chain are the primary targets for malformed batch fuzzing. A harness that constructs raw batch buffers with mismatched type IDs, invalid sizes, or overlapping cross-adapter surface entries — and submits them via direct syscall to bypass the user-mode size check in CommitToKernel — has the highest probability of hitting unvalidated code paths in the kernel.

Conclusions & Next Steps

The reconnaissance phase is complete. No exploitable vulnerability was identified — but that is not the conclusion of the research, it is the baseline. The structural understanding built here is the prerequisite for any meaningful fuzzing. You cannot fuzz what you don't understand.

What was established:

  • SSNs for NtDCompositionCommitChannel (0x111F) and NtDCompositionCreateChannel (0x1124) extracted and verified on Build 19045.6466
  • Full user-mode commit pipeline traced through all three phases with critical vtable offsets and field offsets documented
  • Kernel handler NtDCompositionCommitChannel partially reversed — pointer probing, handle lookup, and batch dispatch logic fully understood
  • Batch dispatcher FUN_1c005ee20 annotated with fast path, full path, command serializer chain, and ETW hooks identified

Planned next steps, in order:

  1. Direct Syscall Harness — Build a harness invoking NtDCompositionCreateChannel then NtDCompositionCommitChannel directly, bypassing dcomp.dll entirely and with it, the user-mode batch size integrity check.
  2. Malformed Batch Fuzzing — Target edge cases in FUN_1c005ee20: mismatched batch size counters, invalid command type IDs (especially 0x34 and 0x0C), and overlapping cross-adapter surface entries.
  3. Full PDB Reverse of win32kbase.sys — Load with symbols into Ghidra to fully reverse FUN_1c005f594 (batch compositor dispatch) and FUN_1c005f78c (batch record allocator).
  4. Live Kernel Instrumentation — WinDbg hardware breakpoint on FUN_1c005ee20 to capture live batch structure layouts during normal DWM operation.
  5. Error Path Audit — Specifically audit the NTSTATUS | 0x10000000 fallback mapping. Error codes suppressed here may indicate kernel code paths that were not expected to be reachable from user mode.
// TAKEAWAY
The Win32k graphics syscall range (0x11xx) is a historically fertile attack surface. DComp specifically adds a stateful channel abstraction on top — a serialized batch protocol that the kernel must deserialize. Protocols that deserialize untrusted input in Ring 0 are where kernel bugs live. The work here establishes the map. The fuzzer finds the gaps in it.
